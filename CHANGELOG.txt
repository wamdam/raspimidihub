RaspiMIDIHub Changelog

User-visible changes. Short tagged one-liners, no code or commit
hashes. Tags: Added, Improved, Fix, Changed, Removed.


2026-05-17 — Version 4.4.4

Three fixes around drop-buttons and the spectator view. The
drop-button one is the most consequential: fade-mode mute changes
were landing on top of the new bar's first downbeat instead of
just before it, which audibly played one bar in the old mute
state. The other two are spectator gaps — the pattern long-press
menu and the device-detail panel's test sender / MIDI monitor
now make it across the wire.

- Fix: drop-button snapshots in **Fade mode** now actually arrive at the destination synth before the new bar starts. Fade slots previously left their discrete-cell CCs (mute buttons, XY pads) un-scheduled and emitted them on the bar boundary itself, so the first bar's downbeat played in the OLD mute state. Now both Hard and Fade slots pre-schedule the full snapshot via the ALSA queue, 8 ms ahead of the bar.
- Fix: a rare edge case where the clock-period EMA wasn't ready at press time (only one MIDI clock tick observed) lost the snapshot CCs entirely — Hard slots fell through to a state-only flip that never sent the CC. Now the slot tracks whether pre-schedule succeeded and falls back to an immediate send at fire moment if not.
- Fix: the long-press menu on a pattern slot (Euclidean / Arpeggiator / Tracker) now mirrors to spectators. The menu's open-slot state was the last piece of pattern-bank UI still held in component-local state instead of `useSharedUiState`, so a phone long-press never made it across the spectator broadcast.
- Fix: the CC test sender and MIDI monitor inside an instrument / plugin's device-detail panel now show up on the spectator when the source opens them. Their open/closed toggles were the last component-local pieces of that panel; the spectator just didn't see them flip. The monitor's event list flows through the same per-component SSE on the spectator's side, so the displayed events match the source.


2026-05-17 — Version 4.4.3

A reflow-vs-click bug in the Add/Edit Mapping overlay: tapping the
**Channel Remap** type radio dropped the panel from ~650 px to
~350 px between the touch starting on the radio and the browser
synthesising the click — and the panel's tap-outside-to-dismiss
handler then matched the gray backdrop above the now-shorter panel
and closed itself. The handler now latches the *press-down* target,
so only a tap that actually started outside the panel dismisses it.

- Fix: tapping the **Channel Remap** type radio in the Add/Edit Mapping panel no longer closes the panel.
- Fix: type-radio pills (and any other future PluginRadio) now opt out of the panel's swipe-down gesture, matching the convention every other interactive control already followed.


2026-05-17 — Version 4.4.2

Two falsy-fallback bugs where the value `0` was getting swapped
for the plugin author's default because of Python's `value or
default` idiom (0 is falsy). Both surfaced as a control whose
zero position visibly disagreed with what came out.

- Fix: CC LFO **Depth = 0** now produces the flat line at center it advertises, instead of silently jumping back to the 127 default and emitting full-swing modulation. Same fix on CC LFO **Center** so a zero-center setting stays at zero instead of snapping to 64.
- Fix: MIDI Delay **Vel Decay = 0** now means "every echo at full velocity" (no decay), instead of falling back to the 20% default.


2026-05-17 — Version 4.4.1

Hot-fix for a CC LFO config panel that wouldn't accept input on
anything other than the wheels. A `</canvas>` tag had gone
missing in the light-theme tokenisation pass; htm / Preact then
parsed the trailing `2s` duration label as canvas fallback
content, the resulting DOM nested unintended siblings inside the
`<canvas>`, and the malformed subtree intercepted pointer events
on every Radio / Button / Fader in the panel.

- Fix: CC LFO (and any future plugin shipping a `DisplayScope`) renders its scope inside a properly-closed `<canvas></canvas>` again. Faders, radios, buttons, and the Sync toggle all respond to taps and drags. Only CC LFO was affected — it's the sole built-in plugin using a scope display.


2026-05-17 — Version 4.4.0

Two ways to make mappings express more with fewer rules, and the
long-press "Bind CC" popup is now reachable from regular plugin
config panels, not just play and controller surfaces. Plus a pair
of fixes for problems that bit the previous release: an Install
button that wedged after any prior install was interrupted, and a
class of matrix glitches where icons and labels drifted onto the
wrong rows after devices were added or removed.

- Added: Note → Note and Note → CC source-note wheels gain an **Any** tick at the top. Persisted as a null sentinel, dispatched as a wildcard. Folds an entire keyboard onto a single destination pitch (one-finger sampler trigger) or fires a CC on every keypress regardless of which key was hit.
- Added: Note → CC mappings have a **Value Source** toggle — **Fixed** (current behaviour, On value sent on every Note On) or **Velocity**. In Velocity mode the live note-on velocity rides through as the CC value; the Off value still applies on release so the CC has a defined idle state. Combined with **Src Note = Any** this turns the whole keyboard into a velocity-to-CC expression pedal.
- Added: More plugin-config controls accept CC bindings through the same long-press → "Bind CC" popup that has always worked on play surfaces. Newly bindable in the config panel: MIDI Delay's Vel Decay; CC LFO's Center; CC Smoother's Smoothing; Chord Generator's Added-Note Vel %; Clock Divider's Divide-by; Master Clock's BPM; Note Splitter's Lower / Upper Transpose; Scale Remapper's Root; Velocity Equalizer's Velocity / Min / Max. The dispatcher, persistence and Learn flow were already unified — only the plugin authors' `default_cc=<N>` declarations were missing.
- Fix: MIDI Learn is now reachable when *editing* an existing mapping, not just when creating one. A hardware controller's channel or CC# can change without retyping every other field on the form.
- Fix: Install button no longer fails permanently after a prior install was killed mid-flight. The install script runs `dpkg --configure -a` defensively before `apt-get install`, so a half-applied dpkg state reconciles automatically instead of surfacing as the cryptic "E: dpkg was interrupted, you must manually run 'dpkg --configure -a'" error on every subsequent attempt. No-op on a clean system.
- Fix: Matrix row headers no longer show the wrong icon, a stale plugin SVG, or a missing label after devices are added or removed. Rows now carry a stable key derived from the port identity so Preact tracks them by device instead of by position; component state (notably PluginIcon's async-fetched SVG) no longer leaks between rows when the input list reshuffles. The hardware / Bluetooth icons are real components instead of shared module-level vnode constants, and a stale `transition: max-width` rule left over from the deleted scroll-aware shrink animation is gone.
- Removed: UI Demo plugin — developer-only showcase of every UI control type, never documented in the manual, not useful in production configs.


2026-05-17 — Version 4.3.0

A non-intrusive way to mirror any connected device's UI into a
browser tab or OBS Browser Source. Useful for showcase videos
where you want the audience to see exactly what the phone is
doing — including touches — but with crisp text and full
resolution at any output size. No video encoding on the device:
the chosen browser re-renders the app natively, driven by a
small per-source broadcast of UI state. Source devices stay
dormant until at least one spectator opens the URL.

- Added: Spectator mirroring — open `/?spectate=<conn_id>&touches=1` to render the chosen device's UI live. Mirrors viewport, scroll position (vertical .main + horizontal matrix, plus the device-detail panel), the current page/route, layout density, theme, every popup (context menu, CC-binding popup with live wheel values, Cell-binding popup, Plugin Control Mappings rows), and the bottom-bar toast / Save Config button state. A touch-ripple overlay (`&touches=1`) paints the source's pointer trail onto the mirror in real time.
- Added: Spectator presentation knobs — `&frame=1` wraps the mirror in a phone bezel, `&chroma=#hex` paints a chroma-key-friendly backdrop around it (magenta / green / any colour), `&tilt-x=<deg>` / `&tilt-y=<deg>` rotate the framed device in 3D. A floating control panel on the spectator URL has sliders, a colour picker, and a Copy URL button; drag anywhere on the background to live-adjust the tilt. The control panel auto-fades on inactivity so OBS captures stay clean.
- Added: Settings → Spectator mirroring sub-section — set this device's name (shown in other tabs' picker), copy its spectator URL, and see a live list of other connected clients to mirror.
- Improved: Architecture — the feature lives in its own server module (`spectator.py`) and client folder (`static/lib/spectator/`), plugged into the core via three small extension points on WebServer (SSE filter, disconnect hook, subscribe extension). web.py and api.py contain zero spectator-specific code outside their import lines. See manual chapter 4 (System Architecture) for the integration points and the opt-in conventions developers need to know.


2026-05-16 — Version 4.2.0

A light theme, and the design-token system that makes more themes
cheap to add. Every colour the UI paints now resolves through a
named CSS variable; theme files override the variables they care
about and leave the rest to the dark default. Adding a third
theme costs one CSS file and one manifest row.

- Added: Light theme — Settings → Display → Theme picker. Bright daytime look across surfaces, controls and play pads; wheels become pale brushed-aluminium with dark values and a subtle rolling-cylinder shadow, knobs go white with dark indicators and a pink LED ring, faders ride pure-white tracks with pale chrome thumbs, drop buttons sit on pale neutral / warm-pale idle backgrounds, sequencer pads use a grey-pink hierarchy (grey wash for plain "on" steps, pink only for accented / beat steps) plus dashed pink outline for Euclidean's algorithm-suggested cells.
- Added: Dark theme — explicit selection (the previously hard-coded look is now one of two installed themes). First-time visitors with no saved preference inherit their OS's `prefers-color-scheme`; the choice persists in browser localStorage and updates the PWA status-bar colour to match.
- Added: `themes/_tokens.css` — single source of truth for every colour the UI paints. Three semantic groups (semantic palette, component tokens, decorative / physical-object tokens) with comments documenting each token's role.
- Added: `themes/manifest.json` + `lib/theme.js` runtime — manifest lists every installed theme; `token()` helper lets canvas-painting surfaces (Display scope, curve editor, drop-button ring segments) read the live token values so they reskin too.
- Improved: Style.css fully tokenised — the ~156 hardcoded hex / rgba literals across the stylesheet have been replaced with `var(--…)` calls. Visible change in dark mode: none. Future-theme cost: one file.
- Improved: JS-side colour literals in knob.js, display.js, curveeditor.js, dropbuttonrow.js, stepeditor.js and matrix.js routed through the same token vocabulary via inline `style="…: var(--…)"` for SVG and the `token()` helper for canvas paint.


2026-05-16 — Version 4.1.0

User-bindable MIDI CC for every plugin control and every controller cell. Long-press (touch) or right-click (mouse) opens a popup with the current binding, a MIDI Learn button, manual Channel + CC wheels, Reset to factory, and Clear. Settings gains a hub of sub-pages with a new Plugin Control Mappings table listing every CC binding across every instance.

- Added: User-bindable MIDI CC on Wheel / Knob / Fader / Radio / NoteSelect / Button / XYPad. Plugin authors opt in by setting `default_cc` on the param dataclass; users override per instance via long-press → popup. Channel (Any / 1..16) + CC (0..127), MIDI Learn captures the next inbound CC on any routed source with a 30 s timeout, Reset to default restores the plugin author's choice, Clear is durable across reboots. Same-CC collisions inside one instance are flagged but allowed — one hardware knob can drive multiple controls on purpose.
- Added: Long-press cell binding on the Controller page. Each cell's symmetric (channel, cc) pair (touch emits, hardware mirrors) is editable from the cell itself; XY pads get an axis-split popup with a separate MIDI Learn for X and Y. Same popup is reachable by tapping any row in Settings → Plugin Control Mappings.
- Added: Plugin Control Mappings sub-page in Settings — flat editable table of every CC binding across all plugins AND every controller cell. Rows route by kind: plugin params open the CcBinding popup, controller cells open the CellBinding popup. Live cc_map_changed / plugin-changed SSE keeps the table in sync as bindings move.
- Added: Settings hub + sub-page navigation. Sys Info / Network / MIDI / Display / Update / Plugin Control Mappings are now distinct sub-pages under a `< Settings / <title>` back-bar. Per-tab sub-state remembers the last sub-page across bottom-nav switches. Reload App + Reboot Pi moved into Sys Info.
- Improved: Plugin HELP popovers and the matrix-side device-detail panel no longer list per-plugin CC numbers — discovery moves to long-press. Appendix A still carries the factory default for each control.
- Changed: Controller cells' (channel, cc) + MIDI Learn editor is gone from the device-detail panel. The panel keeps name rename, Button On / Off + swap, and XY pad spring config — the bits that aren't bindings.
- Changed: Plugin instance rename via the device-detail panel now broadcasts `plugin-changed` SSE; listeners (Plugin Control Mappings, controller picker) refresh live instead of waiting on a manual refresh.
- Removed: Per-plugin class-level `cc_inputs` dict. Defaults now live on the param dataclass via `default_cc=N`; runtime mapping lives in each instance's `cc_map`. Existing configs migrate transparently — missing `cc_map` falls back to the factory seed.


2026-05-15 — Version 4.0.0

A bigger release than the patch-level cadence the 3.1.x series ran on. Headline: the Euclidean play-surface plugin, an 8-slot pattern bank shared by every play-surface plugin, and a UI pass that touches the matrix, the play surfaces, the device-detail HELP popovers, and tab navigation.

- Added: Euclidean play-surface plugin — held notes voiced through an evenly-distributed (Bjorklund) step pattern. Three layers: algorithm (Pulses / Steps / Rotate), window wave (Phase / Cycles / Open as a sine threshold mask), and a four-state Step grid (default / FORCE_ON / FORCE_ON+accent / FORCE_OFF) with an algorithm-underlay preview of what the generator would do. Internal Scale + Root (9 scales including chromatic pass-through), Tune Spread + Snap (free / octaves / 5ths+oct.), Jitter, Fade In / Fade Out. Full block CC 70..88 (skipping CC 84) covers every play-surface knob; CC 74 / CC 75 stay Arp-consistent. Add via Add → Play.
- Added: Pattern bank on all three play-surface plugins (Tracker, Arpeggiator, Euclidean) — 8 P1..P8 slots, each carrying a snapshot of every play-surface param. Tap to switch immediately on the Arp / Euclidean (held notes + sustain persist across the switch); the Tracker keeps its existing queue-to-bar-boundary behaviour. Edits to any play-surface knob auto-write back to the active slot (live-working-slot semantics). Long-press a slot for a Paste-from-current / Reset-to-default menu. A Setup-group Ctrl Ch reserves a MIDI channel for 8 individually MIDI-Learnable trigger notes, one per slot.
- Added: Arpeggiator `chord` pattern — every held note fires simultaneously each step, with the per-step offset / accent / gate applied to the whole burst. Octaves > 1 doubles the chord up. Stored as a new index appended to _PATTERN_OPTIONS so existing configs keep their pattern selection.
- Added: StepEditor override_mode — four-state head cycle (default → FORCE_ON → FORCE_ON+accent → FORCE_OFF → default) with an algorithm-underlay tint on default cells. Used by the Euclidean plugin; the Arpeggiator keeps its three-state cycle.
- Added: Scroll-assist buttons — round accent-coloured FABs in the top-right (▲) and bottom-right (▼) of any overflowing page. Each tap scrolls 200 px in that direction; the buttons appear only when there's content past the viewport edge. Toggle in Settings → Display → "Scroll-assist buttons" (per-device, default on).
- Added: Settings → Display → "(this device only)" hint on the section heading covers all four toggles (MIDI activity bar, Knob/wheel tick sounds, Scroll-assist buttons, Layout density) — every Display preference is browser-local.
- Improved: Bottom-nav tab clicks remember the last-viewed sub-state per tab — leaving Play with the Euclidean open and bouncing through Routing returns you to the Euclidean, not the first surface. Open device-detail panel on Routing and selected Controller instance remembered the same way. Per-device, stored in the browser.
- Improved: Routing matrix row-header column is permanently narrow (~96 px) with middle-truncated labels (Velocity Equalizer 1 → Velo…ar 1) so the matrix gets the room. Tapping a row opens its context menu with the full label at the top.
- Improved: Tracker pattern row, Arp and Euclidean pattern strips share one widget — long-press on any P1..P8 opens an Overwrite-from-current / Reset-to-default menu. Single CSS / single component for all three play-surface plugins. Tracker slots relabelled P1..P8 (was 1..8); "PATTERN" caption removed.
- Improved: Arpeggiator CC table expanded to CC 70..83 (pattern, octaves, steps, rate, gate, accent vel.) so a hardware controller wired for the Euclidean drives the matching Arp knobs identically.
- Improved: Scale catalogue + nearest-in-scale quantiser moved to a shared `raspimidihub.scales` module so the Euclidean plugin and the Scale Remapper share one source of truth.
- Changed: Controller page scrolls vertically again — taller controller layouts no longer clip on short screens. Knob / wheel / fader drags still don't reach the page handler thanks to cell-level stopPropagation, so the horizontal-swipe between instances stays unambiguous.
- Changed: Arpeggiator + Euclidean Setup collapsed to a single "Ctrl Ch" (Off / 1..16) — the same channel reserves the slot-trigger notes AND is the only second-input channel. The dead control_channel ChannelSelect (left over from the removed Trigger Note features) is gone. Setup now reads: Sync / Arp Ch + Ctrl Ch (+ Retrig on Euclidean) / BPM / Pattern Notes.
- Removed: Arpeggiator "Trigger Note" + Base / Euclidean "Pattern Trigger" + Base Setup-group features (Base + contiguous-semitone range that picked Rate / Pattern mode). The 8 per-slot trigger notes under "Ctrl Ch" cover the use case more flexibly; the removed feature was a duplicate path with a less ergonomic shape (single-base only).
- Fix: device-detail HELP popovers reflow to the panel width — source 70-col line wrapping now collapses to natural paragraphs / bullet lists instead of hard-breaking mid-sentence.
- Fix: nested Group inside another Group (Arp / Euclidean Setup → Pattern Notes) now renders its children and honours visible_when. Previously the inner group dropped through to "Unknown" and the 8 learnable trigger notes never appeared even when Ctrl Ch was set.
- Fix: update-check no longer strands the Pi off-AP when NetworkManager is slow — an `nmcli connection reload` taking >5 s used to crash the WiFi switch and cancel the 180 s watchdog that should have brought the AP back. Reload calls now tolerate slow returns, the fallback restores AP on any client-mode error (return-False *or* raise), and the watchdog stays armed until wifi mode is verifiably back to AP.

2026-05-03 — Version 3.0.1

Major release. Bluetooth MIDI, Controllers (Mixer 8 / FX 6 /
Performance 16 / XY 4), drop buttons with bar-quantised fire,
ALSA queue scheduling, USB tethering, in-app updates with
apt-resolved dependencies, per-device clock vetoes, hostapd via
systemd, SysEx Sender plugin. (Numbered 3.0.1 to clear apt's
3.0.0a2 alpha trailer — Debian's comparator treats `3.0.0a2-1`
as newer than `3.0.0-1`.)

Bluetooth MIDI
- Added: BLE-MIDI bridge — paired devices appear in the matrix with a
  Bluetooth icon (separate row + column for IN / OUT). End-to-end
  message coverage: notes, CC, program change, pitch bend, channel /
  poly aftertouch, plus MIDI Clock, Start, Stop, Continue and Song
  Position. SysEx pending.
- Added: Add Device → Bluetooth section — Scan, Connect, Disconnect,
  Forget. MIDI-only filter on by default; "Show all (N other)"
  reveals every BLE peripheral in range.
- Added: Just-Works pairing agent registered automatically. WIDI
  Master and similar peripherals that demand encrypted GATT subscribe
  pair without user interaction.
- Added: Auto-reconnect on service restart for paired BLE-MIDI
  devices.
- Added: Reconnect entry in the matrix context menu for paired-but-
  offline BLE-MIDI devices, alongside Forget.
- Added: BlueZ disconnect detection — when a peripheral goes out of
  range or is powered off, the matrix flips its row/column to offline
  within ~1-2 seconds.
- Added: /var/lib/bluetooth persistence under read-only root — tmpfs
  at boot restored from a tarball on /boot/firmware, snapshot taken
  on every change via inotify so bonds survive immediate power-off
  without a graceful shutdown.
- Changed: BlueZ's built-in `midi` plugin is disabled (systemd drop-
  in, `bluetoothd -P midi`) because it doesn't actually forward GATT
  notifications. We bridge BLE-MIDI in Python instead.

Controllers
- Added: Controller tab in the top nav — fullscreen play surface, swipe / arrow / dropdown to switch between instances, last-viewed remembered
- Added: Mixer 8 — 24 knobs / 8 faders / 16 buttons (CC 16-63 ch 1)
- Added: FX 6 — 18 knobs / 6 faders / 6 buttons (CC 16-45 ch 1)
